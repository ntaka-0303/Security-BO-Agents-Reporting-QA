## PoC ユースケース仕様書：帳票理解・QA エージェント（報告書問い合わせ対応）

### 1. 概要

- **目的**
  - 顧客・営業店・社内からの「各種報告書（取引報告書、残高報告書、年間取引報告書など）に関する問い合わせ」に対して、AI エージェントが回答案を提示し、担当者の調査・回答作成時間を削減する。
- **想定利用者**
  - コンタクトセンター／営業店担当者
  - バックオフィスの問い合わせ対応担当
- **PoC ゴールイメージ**
  - 帳票内容の確認・説明にかかる時間を 30〜60％ 程度削減しつつ、回答品質（正確性・一貫性）を維持または向上できるかを検証する。

---

### 2. 対象業務スコープ

- **対象とする帳票（PoC 第1段階）**
  - 特定口座年間取引報告書
  - 取引報告書（約定ごとのご報告）
  - 残高報告書

- **想定される問い合わせ例**
  - 「このマイナスの金額は何を意味していますか？」
  - 「この残高と、アプリに表示されている残高の違いは何ですか？」
  - 「年間取引報告書のこの欄の数字は、どの取引の合計ですか？」

- **業務フロー上の位置付け**
  1. 顧客からの問い合わせがコール・チャット・メール等で受付
  2. オペレーター／営業担当が、対象帳票を画面に表示または PDF を取得
  3. **オペレーターが AI エージェントに「帳票＋質問内容」を入力**
  4. AI エージェントが回答案＋根拠を提示
  5. オペレーターが内容を確認・補足し、顧客へ回答

---

### 3. インプット／アウトプット定義

#### 3.1 インプット（AI への入力）

- **必須インプット**
  - 帳票データ
    - 形式：PDF、HTML、もしくはそこから抽出した構造化テキスト
  - 問い合わせ内容（自然文）
    - 例：「2ページ目の『株式等譲渡所得の金額』の123,456円は、どの取引の合計ですか？」

- **任意インプット**
  - 過去のやり取り履歴（類似問い合わせの回答例）
  - 顧客の属性（個人／法人、投資経験など）※回答の表現レベル調整に利用

#### 3.2 アウトプット（AI からの出力）

- **回答案（対顧客向け文面案）**
  - 平易な日本語での説明文
  - 例：「お尋ねの123,456円は、2025年中にお取引いただいた株式の売却損益の合計金額です。別紙取引明細の○○〜○○の取引が対象となります。」

- **根拠情報**
  - 帳票上の該当箇所：
    - ページ番号
    - 項目名
    - 可能であれば、該当する行・列の抜粋
  - マニュアル・FAQ の該当部分（あれば）：
    - ドキュメント名
    - 章・項番・抜粋テキスト

- **オペレーター向けメモ**
  - 顧客へ説明する際の補足ポイント
  - 注意喚起（例：税務上の最終判断は税理士等に確認いただく必要がある 等）

---

### 4. 具体的な入出力例（サンプル）

#### 4.1 インプット例（要約）

- 帳票：特定口座年間取引報告書（PDF）
- 質問：
  - 「1ページ目の『譲渡による収入金額（売却代金）』の1,000,000円は、どの取引の合計ですか？」

#### 4.2 アウトプット例（イメージ）

- 回答案：
  - 「『譲渡による収入金額（売却代金）』の1,000,000円は、2025年中に特定口座で行われた株式等の売却代金の合計額です。  
    別紙『年間取引明細』のA社・B社・C社の売却取引（2025年3月1日〜10月15日）の売却代金を合計した金額に相当します。」

- 根拠情報：
  - 帳票該当箇所：
    - 「特定口座年間取引報告書」1ページ目 中央部「譲渡による収入金額（売却代金）」欄
  - 明細該当箇所：
    - 「年間取引明細」2ページ目〜3ページ目の A社／B社／C社の売却行

### 5. 成功指標（KPI）案

- **定量指標**
  - 問い合わせ1件あたりの調査・回答作成時間の平均（PoC前後比較）
  - AI 回答案の「そのまま／一部修正で利用できた」割合
  - 担当者が自力で調査した場合との時間差（ベンチマーク）

- **定性指標**
  - 担当者アンケート（分かりやすさ、有用性、信頼性）
  - 新人・非熟練者でも、一定水準の回答案を出せたか

---

### 6. 制約条件・リスク・前提

- **制約条件**
  - 顧客に直接 AI が回答するのではなく、必ずオペレーターが内容を確認してから回答する。
  - PoC 期間中は、本番顧客情報を利用する場合、情報管理・マスキング等のルールを別途定める。

- **リスク**
  - 帳票構造の解釈誤りにより、誤った根拠を提示する可能性
  - 専門用語の誤解釈により、不正確な説明が生成されるリスク

- **前提**
  - 帳票レイアウト仕様書・項目定義書など、帳票の意味を定義したドキュメントが存在すること。
  - 過去のFAQや標準回答例を一定数用意し、AI に参照させること。

### 7. 業務フロー詳細（文章）

#### 7.1 現行フロー（概略）

1. **問い合わせ受付**
   - 顧客からコールセンター・営業店・チャットなどを通じて問い合わせが入る。
   - 例：「年間取引報告書のこの数字の意味を教えてほしい」など。
2. **帳票の特定・取得**
   - オペレーター／営業担当が顧客情報をもとに、対象帳票（PDF／画面）をシステムから取得。
3. **担当者による内容読解・調査**
   - 帳票の該当箇所を確認し、関連する取引明細・マニュアル・FAQ を参照しながら意味を解釈。
4. **回答案の作成**
   - 社内ルールに沿って、顧客向けの説明文を作成。
5. **必要に応じたエスカレーション**
   - 判断が難しい場合はバックオフィス・専門部署へ問い合わせし、回答内容を協議。
6. **顧客への回答**
   - 電話・メール・チャット等で回答を伝達。

#### 7.2 PoC フロー（AI エージェント導入後の想定）

1. **問い合わせ受付（現行と同じ）**
   - 顧客からの問い合わせを受付。
2. **帳票の特定・取得（現行と同じ）**
   - オペレーターが対象帳票をシステムから取得。
3. **AI エージェントへの入力**
   - オペレーターが AI エージェント画面を開き、
     - 帳票（PDFまたは画面キャプチャ／構造化テキスト）
     - 顧客の質問内容（自然文）
     を入力またはシステム連携で渡す。
4. **AI による解析・回答案生成**
   - AI エージェントが帳票構造と質問内容を解析し、
     - 回答案（対顧客向けの日本語文）
     - 該当箇所の根拠情報（ページ／項目名 等）
   を生成。
5. **オペレーターによるレビュー・編集**
   - 回答案の内容・表現を確認し、必要に応じて修正。
   - 不明点があれば AI に追加質問（「この金額の内訳をもう少し詳しく」など）。
6. **必要に応じたエスカレーション**
   - AI を使っても判断が難しい案件は、従来どおりバックオフィス等にエスカレーション。
7. **顧客への回答**
   - 確定した回答内容を、電話・メール・チャット等で顧客へ伝達。

### 8. 業務フロー図（draw.io ベース）

#### 8.1 図の構成（スイムレーン）

draw.io 上では、以下のスイムレーン構成を推奨：

- レーン1：**顧客**
- レーン2：**コンタクトセンター／営業店担当者**
- レーン3：**AI エージェント**
- レーン4：**バックオフィス（エスカレーション先）**
- レーン5：**帳票システム／基幹システム**

#### 8.2 主要ノードと矢印（例）

- レーン1（顧客）
  - ノード：`帳票を確認`
    - → `問い合わせ実施`
  - ノード：`問い合わせ実施`
    - → レーン2 の `問い合わせを受け付け`

- レーン2（コンタクトセンター／営業店）
  - ノード：`問い合わせを受け付け`
    - → レーン5 の `対象帳票を取得`
  - ノード（レーン5 に配置）：`対象帳票を取得`
    - → レーン2 の `AIエージェント画面を起動`
  - ノード：`AIエージェント画面を起動`
    - → レーン3 の `帳票＋質問内容を入力`

- レーン3（AI エージェント）
  - ノード：`帳票＋質問内容を入力`
    - → `帳票解析・該当箇所特定`
  - ノード：`帳票解析・該当箇所特定`
    - → `回答案＋根拠生成`
  - ノード：`回答案＋根拠生成`
    - → レーン2 の `回答案レビュー・編集`

- レーン2（コンタクトセンター／営業店）
  - ノード：`回答案レビュー・編集`
    - → 条件分岐ノード：`判断できる？`
  - 分岐：`Yes`
    - → `顧客へ回答`
    - → レーン1 の `回答受領`
  - 分岐：`No`
    - → レーン4 の `バックオフィスへエスカレーション`

- レーン4（バックオフィス）
  - ノード：`バックオフィスへエスカレーション`
    - → `詳細調査・回答案作成`
  - ノード：`詳細調査・回答案作成`
    - → レーン2 の `最終回答内容の確認`

- レーン2（コンタクトセンター／営業店）
  - ノード：`最終回答内容の確認`
    - → `顧客へ回答`
    - → レーン1 の `回答受領`

#### 8.3 draw.io での作図ポイント

- スイムレーン図テンプレートを使用し、顧客 → オペレーター → AI → バックオフィス → 帳票システムの順にレーンを並べると、流れが直感的になる。
- AI エージェントに関わるノード（解析・回答案生成）は、レーン3内で色付きの長方形にして PoC 部分を強調。
- 分岐（「AI だけでは判断困難」）はダイヤモンド型で表現し、バックオフィスへのエスカレーション経路を明示。
- 将来 PoC の結果を踏まえ、「顧客への自動回答」パターンを別途破線矢印として追記できるよう、余白を持たせて作図しておくと拡張しやすい。
